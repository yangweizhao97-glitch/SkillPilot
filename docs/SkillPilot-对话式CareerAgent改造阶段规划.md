# SkillPilot 对话式 Career Agent 改造阶段规划

## 1. 改造目标

当前 SkillPilot 更接近一个固定入口的职业分析工作流：用户选择简历和岗位，启动后端任务，再查看执行过程和报告。

下一阶段目标是把产品入口改成一个对话式 Career Agent：

```text
用户自然语言输入目标
  -> Agent 识别意图
  -> 判断缺少哪些资料
  -> 引导用户上传或选择简历 / JD / 报告
  -> 调用已有受控工作流和工具
  -> 在对话中返回进度、结果和下一步建议
```

新的定位不是废掉已有 workflow，而是让 workflow 成为 Agent 背后的执行能力：

```text
前台是 Agent
后台是 Workflow
```

目标界面形态也要随之收敛：

```text
左侧：Agent 对话 / 报告 两个入口
右侧：当前入口对应的主体界面
隐藏能力：新建分析、任务进度、模拟面试、AI 答疑、学习计划、简历 / JD 上传
触发方式：由用户在对话中提出需求，再由 Agent 识别意图并决定调用哪个能力
```

其中：

- `Agent 对话` 是默认入口，点进去后只有一个主对话框。用户开始聊天后，需要保留聊天记录。
- `报告` 是结果资产入口，点进去后只展示历史报告留痕和报告详情。
- 简历上传、JD 上传、学习计划、模拟面试、AI 答疑等能力不再作为固定工作台暴露，而是在对话中由 Agent 通过意图识别、自主判断和追问触发。

报告应该保留为独立资产页，因为它不是普通功能入口，而是后续问答、学习计划、模拟面试和继续优化的上下文来源。

## 2. 阶段一：产品形态重定位

目标：统一项目对外叙事和内部理解。

建议定位：

> SkillPilot 是一个对话式 Career Agent。用户通过一个对话框表达求职准备目标，Agent 负责理解意图、收集资料、调用受控工作流，并生成报告、练习和建议。

本阶段重点：

- 明确“Agent 是用户入口和决策层，Workflow 是后端执行层”。
- 明确“报告是可复用结果资产”，可以作为左侧常驻上下文。
- 移除面向用户的一组固定功能菜单，把功能入口收敛到对话意图识别。
- 保留已有职业分析 workflow，不做推翻式重构。
- 把 README、设计文档、答辩稿中的项目叙事逐步调整为“对话式 Career Agent”。
- 说明现有岗位匹配、简历分析、面试题生成等能力是 Agent 可调用的后台能力。

产出物：

- 新产品定位说明。
- 更新后的项目介绍和答辩口径。
- 前后台职责边界说明。

## 3. 阶段二：前端改成单对话入口

目标：先让用户看到的产品形态从“侧边栏多功能菜单 + 表单 + 工作流面板”变成“左侧双入口 + 右侧对话或报告界面”。

本阶段不要求马上实现复杂意图识别，可以先用轻量规则和现有接口支撑体验。

前端形态建议：

- 登录后默认进入 Career Agent 对话页。
- 左侧只保留两个一级入口：`Agent 对话` 和 `报告`。
- 点击 `Agent 对话` 后，右侧只有一个主对话框，作为所有能力的统一入口。
- 点击 `报告` 后，右侧展示之前生成的报告留痕、报告列表和报告详情。
- 不再把工作台、新建分析、任务、模拟面试、AI 答疑作为一级菜单暴露给用户。
- 支持用户输入自然语言目标，例如“帮我看看这份简历适不适合 Java 后端岗位”。
- 不常驻展示“上传简历 / 上传 JD”的资料工作台；只有当 Agent 判断缺资料时，才在对话中出现上传或选择卡片。
- 支持聊天记录留存，用户重新进入 `Agent 对话` 后能看到历史对话。
- Agent 用消息形式提示缺少的资料。
- 任务进度、工具调用、工作流步骤以对话消息、过程卡片或轻量时间线形式出现，不再把底层 workflow 作为第一视觉重点。
- 报告入口、学习计划、模拟面试入口都可以从对话结果中出现。

已确认的体验要求：

- Agent 对话页需要展示“思考过程摘要”和“工具调用状态”，但不展示原始 chain-of-thought。
- 可展示的过程包括：“正在判断是否需要 JD”“正在检索历史报告”“正在调用报告检索工具”“正在调用简历分析工具”“正在生成面试题”等。
- 当 Agent 判断缺少简历、JD、报告或其他上下文时，必须在对话里出现明确的上传 / 选择卡片，而不是让用户自己去找隐藏入口。
- Agent 对话聊天记录必须落库保存，刷新页面、重新登录或重新进入 `Agent 对话` 后仍然存在。
- Agent 对话页是操作入口，报告页是资产留痕入口，两者不能混成同一个页面体验。

推荐布局：

```text
┌────────────────────┬──────────────────────────────┐
│ Agent 对话          │ Career Agent 对话框           │
│ 报告                │                              │
│                    │ 用户：帮我分析这份 JD          │
│                    │ Agent：正在识别意图...         │
│                    │ Agent：需要 JD，请上传或选择   │
│                    │ Agent：开始调用分析工作流...   │
│                    │ Agent：报告已生成              │
└────────────────────┴──────────────────────────────┘
```

报告页布局：

```text
┌────────────────────┬──────────────────────────────┐
│ Agent 对话          │ 历史报告                      │
│ 报告                │                              │
│                    │ 报告 A                         │
│                    │ 报告 B                         │
│                    │ 报告 C                         │
│                    │ 点击后进入报告详情             │
└────────────────────┴──────────────────────────────┘
```

能力触发方式：

| 能力 | 前端呈现方式 |
| --- | --- |
| 报告 | 左侧常驻结果资产 |
| 新建分析 | 用户说“帮我分析……”后触发 |
| 任务进度 | 对话中显示进度卡片 |
| 模拟面试 | 用户说“开始模拟面试”后触发 |
| AI 答疑 | 用户直接提问后触发 |
| 学习计划 | 用户说“生成学习计划”后触发 |
| 简历 / JD 上传 | Agent 发现缺资料时，在对话中提示上传或选择 |

本阶段可以继续复用：

- 文件上传接口。
- 简历和岗位管理接口。
- 职业分析任务接口。
- SSE 任务进度接口。
- 报告查询接口。

产出物：

- 左侧 `Agent 对话 / 报告` 双入口。
- `Agent 对话` 页的单一主对话框和聊天记录。
- `报告` 页的历史报告留痕和报告详情。
- 对话式上传和资料选择体验。
- 对话中的任务进度展示。

## 4. 阶段三：加入意图识别层

目标：让系统真正从固定流程入口升级为“Agent 根据用户意图选择能力”。

建议新增一个意图识别边界，例如：

- `ConversationAgent`
- `IntentRouter`
- `CareerIntentService`

它负责把用户输入归类成明确意图。

第一批意图建议：

| Intent | 含义 |
| --- | --- |
| `CAREER_ANALYSIS` | 基于简历和 JD 跑完整职业分析 |
| `RESUME_REVIEW` | 只分析或优化简历 |
| `JOB_MATCH` | 判断简历和岗位匹配度 |
| `INTERVIEW_PREP` | 生成面试题或面试准备建议 |
| `MOCK_INTERVIEW` | 进入模拟面试 |
| `LEARNING_PLAN` | 基于报告或目标生成学习计划 |
| `REPORT_QA` | 围绕已有报告继续问答 |
| `GENERAL_CAREER_QA` | 通用求职、学习、面试问答 |

实现策略：

- 早期可以用规则 + 少量 LLM 分类。
- 分类输出必须是结构化 JSON。
- 识别结果需要包含置信度、所需资源、下一步动作。
- 低置信度时不要强行执行，先向用户澄清。

产出物：

- 意图识别接口。
- 意图 JSON Schema。
- 意图到后端能力的路由表。
- 意图识别单元测试和回归样例。

## 5. 阶段四：上下文补全和工作流调用

目标：Agent 不只是识别用户想做什么，还要判断能不能执行、缺什么、下一步应该问什么。

例如用户输入：

> 帮我看看我适不适合这个岗位

Agent 应该判断：

- 当前会话有没有简历？
- 当前会话有没有岗位 JD？
- 用户历史资料中是否已有可用简历？
- 用户历史资料中是否已有可用岗位？
- 是否需要让用户上传文件、选择已有资料，或直接启动分析？

建议新增上下文补全能力：

- `ConversationContext`
- `RequiredResource`
- `MissingResourcePrompt`
- `AgentActionPlan`

Agent action 可以包括：

| Action | 说明 |
| --- | --- |
| `ASK_USER` | 向用户追问缺失信息 |
| `REQUEST_UPLOAD` | 要求上传简历、JD 或项目资料 |
| `SELECT_EXISTING_RESOURCE` | 让用户选择已有简历、岗位或报告 |
| `START_WORKFLOW` | 启动已有职业分析 workflow |
| `GET_REPORT` | 查询已有报告 |
| `START_MOCK_INTERVIEW` | 启动模拟面试 |
| `GENERATE_LEARNING_PLAN` | 生成学习计划 |

关键原则：

- Agent 负责判断和编排。
- Workflow 负责稳定执行。
- Tool Registry 继续作为权限和审计边界。
- 不允许模型自由访问数据库、文件系统或外部服务。
- Agent 调用工具或 workflow 时，需要把可公开的执行过程放进对话中，例如“正在识别意图”“正在检索历史报告”“正在调用简历分析工具”“正在生成面试题”。
- 可以展示工具调用状态、检索来源、工作流步骤和结果摘要，但不展示原始模型思维链。
- 普通 AI 回复流、工具调用事件流、workflow SSE 进度流应该逐步收敛成统一的 Agent 消息事件模型。
- Agent 消息事件模型至少需要支持：用户消息、AI 流式文本、过程摘要、工具调用开始、工具调用完成、工具调用失败、workflow 步骤更新、资源上传 / 选择卡片、报告生成完成。

产出物：

- 上下文补全服务。
- 缺失资料追问机制。
- 意图到 workflow/tool 的执行计划。
- 对话消息和后台任务的关联关系。
- 对话中的工具调用 / 工作流过程卡片。

## 6. 阶段五：多轮记忆和 Agent 感增强

目标：让系统从“套了聊天框的工作流工具”变成更自然的 Career Agent。

能力建议：

- 记住用户求职目标，例如秋招、Java 后端、实习、社招、外企等。
- 记住用户薄弱点，例如项目表达、算法、八股、系统设计、英文面试。
- 保留 Agent 对话聊天记录，而不是只在当前页面临时显示。
- 支持流式输出，让 Agent 回复、工具调用过程和工作流状态以连续事件形式呈现。
- 支持基于历史报告继续追问。
- 支持从报告自然跳转到学习计划或模拟面试。
- 支持用户说“那针对这个弱点出 5 道题”。
- 支持 Agent 主动建议下一步，例如“你现在更适合先补项目表达，再刷并发和数据库题”。
- 迁移现有 AI 答疑 / tutor streaming 能力，让它成为 Agent 对话的底层能力，而不是独立一级入口。
- 参考 Kimi 网页版一类体验：对外展示“正在做什么”和“调用了什么能力”，例如意图识别、资料检索、工具调用、报告生成、学习计划生成等。

后端建议：

- 复用现有 tutor memory 或抽象出通用会话记忆。
- 记忆内容分层：短期会话上下文、长期用户画像、任务级报告产物。
- 重要结论必须可追溯到报告、用户资料或模型生成记录。
- 新增或复用会话表保存 Agent 对话历史，消息需要区分用户消息、助手消息、工具 / 工作流过程消息和资源卡片消息。
- 对话历史不只用于展示，也应该成为下一轮意图识别、上下文补全和报告追问的输入之一。

产出物：

- Career profile / 用户求职画像。
- 会话记忆摘要。
- Agent 对话历史记录。
- 流式输出的 Agent 消息事件。
- 基于报告的多轮问答。
- Agent 主动建议下一步的策略。

风险和边界：

- 不能展示原始 chain-of-thought，只能展示可审计、可解释的过程摘要和工具调用状态。
- 完全隐藏上传入口后，用户可能不知道如何补资料，因此缺资料时必须在对话中给出明确的上传或选择卡片。
- 聊天记录需要落库，否则刷新后丢失，会削弱 Agent 感。
- 现有 tutor/chat streaming 和 workflow SSE 的事件模型不同，迁移时需要统一成 Agent 消息事件，避免前端状态复杂度失控。
- 报告页和 Agent 对话页要保持边界清晰：报告页负责资产留痕，Agent 对话页负责理解需求和触发能力。

这些风险对应的执行约束：

- 过程展示只写“正在做什么 / 调用了什么 / 得到了什么摘要”，不输出模型内部逐字推理。
- 资料补齐必须由 Agent 在对话中主动发起，上传和选择入口以卡片形式出现在当前上下文里。
- 聊天记录必须支持刷新后恢复，不能只依赖前端 state。
- 前端统一消费 Agent 消息事件，不再分别维护普通聊天流、工具调用流和 workflow 进度流三套互不相干的 UI 状态。
- 报告页只做报告列表、报告详情、导出和报告留痕；新的分析、学习计划、模拟面试、问答都从 Agent 对话中触发。

## 7. 推荐实施顺序

建议按以下顺序推进：

1. 先改产品定位和前端入口。
2. 再做意图识别。
3. 再让意图识别调用已有 workflow。
4. 最后做多轮记忆和更强的 Agent 自主性。

不要推翻已有职业分析工作流。现有 workflow 是项目的工程优势，它应该成为对话式 Agent 的“执行层”。真正需要新增的是：

- 一个自然语言入口。
- 一个意图识别层。
- 一个上下文补全层。
- 一个把意图路由到 workflow/tool 的执行计划层。

## 8. 最终形态

最终系统应该呈现为：

```text
用户
  -> 对话式 Career Agent
    -> 意图识别
    -> 上下文补全
    -> 工具 / 工作流路由
    -> 职业分析 workflow
    -> 报告 / 面试 / 学习计划 / 问答
```

一句话总结：

> SkillPilot 的前台体验应该是 Agent，后台能力应该是可审计、可恢复、可测试的 Workflow。
