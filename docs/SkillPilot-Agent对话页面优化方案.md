# SkillPilot Agent 对话页面优化方案

## 1. 优化目标

当前 Agent 对话页面仍然偏向“工作流控制台 / 后台任务面板”：简历、岗位、报告、思考过程、工具调用、工作流进度同时出现在主视图中，用户注意力容易被内部执行细节分散。

本次优化目标是把页面调整成“对话优先的 AI 职业助手工作台”：

```text
用户输入目标
  -> Agent 理解意图
  -> 必要时引导补充资料
  -> 在对话流中展示轻量执行状态
  -> 返回结果摘要、报告入口和下一步操作
```

核心原则：

- 前端体验是 Agent 对话，不是 workflow 控制台。
- 后端 workflow、任务、报告能力继续复用，不重写业务逻辑。
- 工具调用和任务进度只作为对话流中的轻量状态事件出现。
- 不展示原始 chain-of-thought，只展示用户可理解的过程摘要和工具状态。

## 2. 页面信息架构

### 2.1 左侧导航

保留左侧导航，但降低视觉重量：

- SkillPilot 品牌。
- 刷新按钮。
- `Agent 对话`。
- `报告`。
- 用户信息 / 退出。

调整要求：

- 缩窄左侧宽度。
- 删除左侧导航和主内容区之间的空白第二栏。
- 左侧只承担导航作用，不承载报告列表或任务列表。

### 2.2 主内容区

主内容区改成居中的聊天列：

- 最大宽度约 900-1000px。
- 左右保留充足留白。
- 聊天内容是页面主体。
- 输入框固定或 sticky 在聊天列底部。

不要在主内容区长期固定展示大块任务面板、工具面板或日志面板。

## 3. 顶部上下文设计

当前“简历 / 岗位 / 最近报告 / 打开报告”的大横向信息栏需要弱化。

建议改成轻量顶部栏：

```text
简历检查 + AI 求职助手                 当前简历 · 目标岗位 · 查看报告
```

上下文信息展示规则：

- 简历、岗位、报告只显示为紧凑标签或入口。
- 用户点击后再展开详情或选择资源。
- 不用大卡片长期占据顶部空间。
- 没有资源时不强行展示空状态大卡片。

## 4. 聊天消息设计

聊天消息是核心。

### 4.1 普通消息

用户消息：

- 靠右展示。
- 使用浅灰或品牌浅色气泡。
- 不使用大边框卡片。

Agent 消息：

- 靠左展示。
- 带 SkillPilot 图标或简洁头像。
- 正常自然语言回复，不强制放进卡片。

示例：

```text
用户：帮我优化这份简历

Agent：我已经读取了你的简历。接下来会先检查岗位关键词匹配，再给出可直接修改的建议。
```

### 4.2 状态事件

思考过程摘要、工具调用状态、工作流进度不要作为大模块固定展示。

改成聊天流里的 `AgentStatusEvent`：

```text
正在读取简历
已读取简历内容
正在匹配岗位 JD
正在生成优化建议
```

运行中：

- 一行状态。
- loading / spinner。
- 简短描述。

完成后：

- 折叠成一行。
- 例如：`已完成简历分析，共执行 4 个步骤`。
- 点击后再展开详细步骤。

禁止展示：

- 原始模型思维链。
- 长篇内部推理。
- 后台日志式的执行明细默认展开。

## 5. 状态事件去重策略

workflow SSE 可能重复推送相似步骤，不能每条都 append 成聊天消息。

建议使用 `statusKey` 去重：

```ts
statusKey = `${taskId}:${stepName}`
```

处理规则：

- 同一个 `statusKey` 只保留一条状态事件。
- 新事件到来时更新该事件的状态和摘要。
- `RUNNING` 显示 loading。
- `SUCCESS` 折叠为完成状态。
- `FAILED` 显示失败状态和重试按钮。

示例状态模型：

```ts
type AgentStatusEvent = {
  statusKey: string
  taskId?: number
  stepName: string
  label: string
  summary: string
  status: 'RUNNING' | 'SUCCESS' | 'FAILED'
  details?: string[]
  collapsed: boolean
}
```

这样可以避免聊天区被 workflow 原始事件刷屏。

## 6. 文件上传与资料识别

上传入口不建议做成固定工作台。

推荐两类入口：

### 6.1 输入框左侧附件按钮

输入框左侧放 `+` 或附件按钮：

```text
[ + ] 输入你的目标...                         [发送]
```

点击后出现小菜单：

- 上传简历。
- 上传 JD。
- 上传项目材料。
- 选择已有简历。
- 选择已有岗位。
- 选择已有报告。

### 6.2 对话中的资料补齐卡片

当 Agent 判断缺资料时，在聊天流中出现上传 / 选择卡片。

示例：

```text
Agent：我需要一份简历和一个目标岗位 JD 才能分析匹配度。

[上传简历] [选择已有简历]
[上传 JD] [选择已有岗位]
```

## 7. 文件类型识别策略

上传文件后，Agent 应该做资料识别，而不是完全依赖用户手动选择。

建议流程：

```text
用户上传文件
  -> 判断文件类型：简历 / JD / 项目材料 / 其他
  -> 写入当前会话上下文
  -> 结合当前意图决定下一步
```

典型场景：

只上传简历：

```text
我识别到这是一份简历。你想让我先做简历优化，还是补充目标 JD 后做岗位匹配？
```

上传两个文件：

```text
我识别到一个是简历，一个是 JD。是否开始分析岗位匹配度？
```

识别不确定：

```text
这两个文件都更像简历。请确认哪一个是目标岗位 JD。
```

实现建议：

- 保留用户显式选择文件类型的能力。
- 后端基于 `fileType` 和内容关键词做二次判断。
- 不确定时用确认卡片让用户选择。

## 8. 结果卡片设计

结果不要做成后台报告大面板，而是作为对话流中的 `ToolResultCard`。

示例：

```text
岗位匹配度：78%

优势：
- Java / Spring Boot
- Redis / MySQL
- 高并发订单项目

待补强：
- 项目量化指标
- 岗位关键词
- 业务表达

[查看完整报告] [继续优化] [生成面试题]
```

使用规则：

- 普通回复不用卡片。
- 结构化结果、报告、确认操作才用卡片。
- 卡片要紧凑，不要大面积浅色背景和重复边框。

## 9. 任务状态设计

需要覆盖以下状态：

| 状态 | 展示方式 |
| --- | --- |
| 执行中 | 一行状态 + loading，例如“正在分析简历” |
| 等待用户确认 | 确认卡片，例如“是否基于该岗位生成面试题？” |
| 执行完成 | 结果摘要 + 下一步操作 |
| 执行失败 | 简短失败原因 + 重试按钮 |
| 已生成报告 | 报告卡片 + 打开报告按钮 |

刷新页面后：

- 需要恢复聊天记录。
- 需要恢复已生成报告入口。
- 不需要恢复执行中的 SSE 任务状态。
- 刷新后执行中任务可以视为中断，用户可重新发起。

## 10. 推荐前端渲染单元

建议抽成以下组件：

```text
ChatMessage
AgentStatusEvent
ToolResultCard
ReportCard
ChatComposer
ResourceRequestCard
```

职责说明：

- `ChatMessage`：普通用户或 Agent 消息。
- `AgentStatusEvent`：轻量运行状态、折叠工作流状态。
- `ToolResultCard`：岗位匹配、简历分析、优化建议等结构化结果。
- `ReportCard`：报告生成完成后的摘要和打开入口。
- `ChatComposer`：底部输入框、附件上传、快捷指令、发送按钮。
- `ResourceRequestCard`：缺资料时的上传 / 选择卡片。

## 11. 视觉原则

减少：

- 大面积浅色卡片。
- 虚线框。
- 绿色边框堆叠。
- 固定展示的后台状态面板。
- 多个系统状态模块同时占据聊天区。

保留：

- SkillPilot 绿色品牌色。
- 简洁图标。
- 清晰对话层级。
- 必要结果卡片。
- 报告页作为独立资产入口。

视觉目标：

```text
用户看到的是：Agent 正在帮我做到哪一步
而不是：后台 workflow 原始日志正在如何执行
```

## 12. 实施顺序

建议按以下顺序实现：

1. 收窄左侧导航，删除中间空白栏。
2. 主区改成居中聊天列。
3. 顶部上下文改成轻量标签。
4. 抽出 `ChatMessage` 和 `ChatComposer`。
5. 把 `PROCESS / TOOL_STATUS / WORKFLOW_STATUS / REPORT_READY` 渲染成 `AgentStatusEvent` 或 `ReportCard`。
6. 实现状态事件 `statusKey` 去重和完成后折叠。
7. 把上传入口移动到输入框左侧附件按钮。
8. 保留缺资料时的聊天内 `ResourceRequestCard`。
9. 把报告生成、学习计划、模拟面试入口做成对话流中的结果/动作卡片。
10. 做场景测试：缺资料、上传简历、上传 JD、完整分析、报告生成、报告追问、学习计划、模拟面试。

## 13. 验收标准

- 页面第一视觉是聊天，不是任务面板。
- 左侧只有导航，不承担工作台职责。
- 简历、岗位、报告上下文不占用大面积空间。
- 状态事件不刷屏。
- 工具调用和 workflow 进度融入对话流。
- 报告生成后在聊天中出现入口，同时报告页保留资产留痕。
- 刷新后聊天记录和已生成报告可恢复。
- 刷新后执行中任务不要求恢复。
- 不展示原始 chain-of-thought。
