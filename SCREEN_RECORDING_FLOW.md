# SkillPilot 三面项目答辩与录屏讲稿

这份稿子面向后端三面和项目录屏。目标不是把所有功能念一遍，而是让面试官在 10 到 15 分钟内形成三个判断：

1. 这是一个真实跑通的后端系统，不是套壳聊天页面。
2. 我能把不稳定的大模型能力放进可控、可追踪的工程链路。
3. 我清楚当前方案的边界，也知道生产化应该先补什么。

答辩时始终围绕一条主线：

```text
用户在对话中提出目标
  -> Agent 识别意图并补齐上下文
  -> Workflow 异步执行确定性步骤
  -> Agent 通过受控工具读取证据
  -> LLM 生成结构化结果
  -> Schema 和业务规则校验
  -> 结果、日志、报告落库
  -> SSE 持续向前端推送状态和文本
```

不要从依赖列表、数据库表或功能菜单开始讲。先讲问题和结果，再讲架构判断，最后用代码、日志和数据库证明。

## 1. 推荐答辩顺序

### 1.1 标准 12 分钟版本

| 时间 | 内容 | 目标 |
| --- | --- | --- |
| 0:00–0:40 | 项目定位和个人职责 | 让面试官马上知道项目解决什么问题 |

| 2:30–4:00 | 整体架构与核心数据流 | 建立后续技术讨论地图 |
| 4:00–6:30 | 难点一：Agent 与 Workflow 分层、异步任务 | 展示后端架构能力 |

| 8:30–10:30 | 难点三：工具权限、输出治理、审计 | 展示安全和可靠性意识 |
| 10:30–11:30 | 数据库、日志和 SSE | 证明可观测、可排障 |

### 1.2 时间不足时的 5 分钟版本

1. 项目定位和职责。
2. 一条完整请求链路。
3. 最关键的架构判断：前台 Agent，后台 Workflow。
4. 当前边界：`@Async` 不是可靠任务队列，Schema 不保证事实正确。

学习计划、模拟面试、Tutor、PDF、公共面经、LangGraph 和 MCP 都放到追问环节。

## 2. 开场怎么讲

### 2.1 30 秒开场

> SkillPilot 是一个对话式 Career Agent。用户不需要理解系统有哪些功能，只要在一个对话框里说出目标，例如“帮我判断这份简历是否适合 Java 后端岗位”。Agent 会识别意图、判断缺少的资料，并调用后端职业分析工作流，最终生成岗位匹配、简历建议、面试题和报告。
>
> 我主要负责后端主链路，包括文档解析和私有 RAG、Agent 与 Workflow 编排、受控工具调用、模型结构化输出校验、异步任务、SSE 事件流、执行日志和报告聚合。这个项目的重点不是接入一个模型接口，而是把模型放进可控、可追踪的后端系统。

### 2.2 90 秒项目背景

> 传统求职工具通常是多个割裂入口：简历分析、岗位匹配、面试题和学习计划各做一块。用户需要先理解产品能力，再决定点哪个功能。
>
> 我把入口收敛成一个对话式 Agent。用户只负责表达目标，Agent 负责意图识别、上下文补全和能力路由；真正稳定执行岗位匹配、简历分析、面试题生成和报告聚合的，仍然是后端 Workflow。
>
> 这里有一个重要设计：前台是 Agent，后台是 Workflow。Agent 适合处理模糊意图和多轮交互，Workflow 适合处理确定步骤、状态推进、失败记录和结果落库。这样既保留了 Agent 的自然交互，也没有把核心业务完全交给不可预测的模型。

### 2.3 一句话技术链路

> React 前端通过 REST 和 SSE 调用 Spring Boot；用户资料经过 Tika 解析、chunk 和 embedding 后进入 PostgreSQL + pgvector；Career Agent 做意图识别和上下文补全，后台任务通过 afterCommit + `@Async` 执行；每个 Agent 只能通过受控工具读取证据，模型结果经过 JSON Schema 和业务校验后落库，最终由报告服务聚合展示。

## 3. 先演示什么，怎么讲

录屏或现场演示不要现场等完整模型任务跑很久。准备一组脱敏的简历、JD 和一份已完成报告，同时保留一次可实时触发的短请求。

### 3.1 演示前准备

- 后端和前端已启动，健康检查为 `UP`。
- `.env` 已正确加载，但屏幕中不展示 API Key。
- 准备一份脱敏简历和一份脱敏 JD。
- 准备一个已经完成的任务，用于展示报告和日志。
- 数据库客户端只保留只读查询，不展示简历全文。
- 浏览器提前登录，避免把答辩时间耗在注册上。

### 3.2 第一步：展示统一对话入口

操作：

1. 打开 Agent 对话页。
2. 输入：“帮我分析这份简历是否适合 Java 后端岗位。”
3. 点击发送。

讲法：

> 用户消息会立即显示，不需要等待整个请求结束。后端通过 `/api/career-agent/plan/stream` 返回 Agent SSE 事件：先表示消息已接收和正在规划；如果是通用问答或报告追问，模型文本会通过 `AGENT_DELTA` 逐段返回；如果需要执行职业分析，则会创建后台任务，再切换到任务 SSE。

这里要强调：

- 用户消息是前端乐观回显，不会因为后端耗时而“点击后没反应”。
- 主 Agent 文本流和后台任务进度流不是一回事。
- 对外只展示“正在识别意图”“正在检索资料”等过程摘要，不展示模型原始思维链。

### 3.3 第二步：展示缺少资料时的上下文补全

如果当前会话没有简历或 JD，Agent 会返回上传/选择卡片。

讲法：

> Agent 不会因为用户少传一个文件就直接失败。它先识别意图，再计算 requiredResources 和 missingResources。缺少资料时返回 `ASK_USER`，由对话界面展示已有资源选择和上传卡片；资料齐全后才允许进入 Workflow。

这一步体现的不是“做了一个上传组件”，而是：

- 意图识别；
- 用户上下文解析；
- 资源所有权校验；
- Agent 决策与确定性工作流之间的边界。

### 3.4 第三步：展示后台任务和 SSE 进度

资料齐全后启动分析。

讲法：

> 后端先创建 `agent_tasks` 记录并返回 taskId，不在 HTTP 请求里等待多个模型调用。事务提交后，通过 afterCommit 启动异步执行。前端订阅 `/api/career-tasks/{taskId}/events`，持续接收读取资料、检索上下文、岗位匹配、简历分析、面试题和报告聚合等事件。

如果任务正在运行，指出：

- 当前步骤；
- 百分比；
- 用户可读摘要；
- 失败后重试入口。

不要把所有技术日志直接塞给用户。用户界面展示业务进度，开发排障再查看技术日志。

### 3.5 第四步：展示报告

打开一份已完成报告。

讲法：

> 最终报告不是让模型重新把所有内容写一遍，而是聚合岗位匹配、简历分析和面试题等已经校验并落库的结构化产物。这样可以减少二次生成造成的事实漂移，也方便追溯每一部分来自哪个 Agent 和哪些证据。

重点展示：

- 匹配结论；
- 优势与差距；
- 简历修改建议；
- 面试题；
- 引用或来源；
- 报告版本和状态。

### 3.6 第五步：用日志证明系统真实执行

讲法：

> 对用户我展示的是简化进度；对开发和排障，我通过 traceId 串联任务、Agent 执行和工具调用。这样能回答：任务停在哪一步、调用了哪个工具、模型耗时多少、是否触发重试、失败原因是什么。

如果时间有限，展示一条 `agent_execution_logs` 和一条 `tool_call_logs` 即可，不要滚动大量日志。

## 4. 架构图怎么讲

推荐按职责边界讲，不要按框架名念技术栈。

```text
┌──────────────────────────────────────────────┐
│ React：对话、资源卡片、报告、SSE 增量渲染     │
└──────────────────────┬───────────────────────┘
                       │ REST / SSE
┌──────────────────────▼───────────────────────┐
│ Spring Boot HTTP 边界                         │
│ JWT、参数校验、统一错误、CurrentUser、traceId │
└───────────────┬──────────────────────────────┘
                │
┌───────────────▼──────────────────────────────┐
│ Career Agent 决策层                           │
│ 意图识别、上下文补全、资源选择、能力路由       │
└───────────────┬──────────────────────────────┘
                │
┌───────────────▼──────────────────────────────┐
│ Workflow 执行层                               │
│ 异步任务、状态推进、Agent 调度、失败记录       │
└───────┬───────────────┬──────────────────────┘
        │               │
┌───────▼────────┐ ┌────▼─────────────────────┐
│ 受控工具层      │ │ LLM 输出治理              │
│ 权限、scope、审计│ │ Schema、repair、业务校验  │
└───────┬────────┘ └────┬─────────────────────┘
        │               │
┌───────▼───────────────▼──────────────────────┐
│ PostgreSQL + pgvector                         │
│ 用户、文档、chunk、任务、日志、产物、报告       │
└──────────────────────────────────────────────┘
```

口述：

> 我把系统分成五层。HTTP 层负责鉴权和输入边界；Career Agent 负责理解用户想做什么；Workflow 负责稳定执行；工具层控制模型能读取什么；输出治理层控制模型结果能不能进入业务系统。PostgreSQL 是当前事实来源，Redis 还没有承担核心一致性职责。

## 5. 核心难点一：为什么是“Agent + Workflow”，而不是全交给模型

### 5.1 先给结论

> Agent 负责处理不确定性，Workflow 负责保证确定性。

### 5.2 展开讲法

Career Agent 的输入是自然语言，用户可能表达得不完整：

- “帮我看看这个岗位。”
- “那针对这个弱点出几道题。”
- “根据之前的报告安排两周学习计划。”

这类请求需要：

- 识别意图；
- 读取会话历史和职业画像；
- 判断需要简历、JD 还是报告；
- 决定追问、上传、查询报告还是启动任务。

但岗位匹配、简历分析、面试题和报告聚合是确定步骤，应该由 Workflow 控制顺序和状态，而不是每次都让模型自由规划。

因此当前设计是：

```text
Career Agent
  -> 识别 CAREER_ANALYSIS / RESUME_REVIEW / JOB_MATCH 等意图
  -> 判断 missingResources
  -> 选择 ASK_USER / START_WORKFLOW / GET_REPORT 等动作

Workflow
  -> MATCHING_JOB
  -> ANALYZING_RESUME
  -> GENERATING_QUESTIONS
  -> GENERATING_FINAL_REPORT
  -> SUCCESS / FAILED
```

### 5.3 这个设计解决了什么

- 用户入口更自然，不需要理解后台功能菜单。
- 核心业务步骤可预测、可测试。
- 每个步骤可以单独记录状态、耗时和错误。
- 模型失败不会让整个系统完全失去状态。
- 后续可以替换规划层，而不推翻用户、文件、权限、任务和报告模块。

### 5.4 LangGraph 的准确定位

不要说“项目主链路依赖 LangGraph”。

正确表达：

> 当前稳定主链路是 Spring 本地工作流。LangGraph 是可选的规划适配层，可以返回执行计划，但计划仍要经过服务端策略校验；不可用或计划不合法时可以回退到默认 Workflow。鉴权、工具执行、Schema 校验和结果落库不会交给 LangGraph。

## 6. 核心难点二：异步任务、事务边界和失败恢复

### 6.1 为什么不能同步执行

一次完整职业分析包含多次检索和模型调用，耗时不稳定。如果全部放在同步 HTTP 请求里：

- 容易超时；
- 用户只能盲等；
- 请求中断后难以恢复现场；
- 无法清晰展示步骤；
- 失败时只剩一个笼统错误。

### 6.2 当前实现

```text
创建任务
  -> 校验 resume/job 属于当前用户
  -> 写 agent_tasks 和初始日志
  -> 事务提交
  -> afterCommit
  -> @Async("careerTaskExecutor")
  -> WorkflowRunner 推进步骤
  -> 日志和结果持续落库
  -> SSE 推送前端
```

为什么使用 afterCommit：

> 如果事务还没有提交就启动异步线程，后台线程可能读不到刚创建的任务，或者看到不完整状态。afterCommit 保证任务和初始日志成为数据库事实之后，再开始执行。

### 6.3 失败怎么处理

- 步骤失败会写入 Agent 执行日志。
- 任务状态更新为 `FAILED`，保存用户可理解的错误摘要。
- 工具调用失败会单独写 `tool_call_logs`。
- 重试接口基于原任务资源创建新任务，原失败任务保留用于审计。

### 6.4 主动说明当前边界

> afterCommit + `@Async` 适合本地和中小规模任务，但它不是可靠持久化队列。进程重启会丢失正在运行的内存任务，多实例还会涉及任务领取和竞争。生产化我会优先引入持久化消息队列或基于任务表的领取机制，并补幂等键、超时、重试退避、死信、限流和告警。

这句话很重要。资深感来自知道方案为什么够用，也知道什么时候不够用。

## 7. 核心难点三：文件如何变成私有 RAG 知识库

### 7.1 完整链路

```text
uploaded_files
  -> Apache Tika 解析
  -> documents 标准化正文
  -> document_chunks 语义切块
  -> embedding
  -> pgvector
  -> vector / keyword / hybrid search
  -> Agent 证据上下文
```

### 7.2 口述版本

> 用户上传简历、JD 或项目材料后，我没有直接把整个文件塞给模型。上传阶段先做大小限制、文件名清理、UUID 存储、用户目录隔离和路径检查。Tika 负责类型识别和正文解析，但我不会把 Tika 描述成完整文件安全方案。
>
> 正文解析后保存为统一 Document，再按标题、段落和 token 窗口切成 chunk。当前默认 targetTokens 是 700，overlap 是 100，每个 chunk 保存来源类型、标题、定位、token 数和正文。随后生成 1024 维 embedding，写入 PostgreSQL 的 pgvector 字段。
>
> 检索层支持向量、关键词和 hybrid search。向量检索负责语义召回，关键词检索负责 Spring Boot、Redis 等技术词的精确命中，最终结果合并去重后作为 Agent 的证据上下文。

### 7.3 为什么选 PostgreSQL + pgvector

> 当前是个人私有知识库和中小规模检索。把关系数据、用户权限、JSON 元数据、正文和向量放在同一个数据库里，事务、备份、审计和开发复杂度更低。数据规模或检索 QPS 明显增长后，再评估 Milvus、Elasticsearch 或独立向量服务。

### 7.4 RAG 的边界

不要说“有 RAG 就不会幻觉”。

正确表达：

- RAG 缩小模型上下文并提供证据。
- Schema 约束结果结构。
- 业务校验检查关键字段和数量。
- 这些措施降低风险，但不能证明每个结论都绝对正确。
- 后续需要固定评测集、召回率评估、rerank 和引用一致性校验。

## 8. 核心难点四：模型为什么不能直接读数据库

### 8.1 核心原则

> 模型负责提出“需要什么”，后端负责判断“能不能取、怎么取、如何审计”。

### 8.2 工具调用边界

当前能力被封装为受控工具，例如：

- `getResume`
- `getJobDescription`
- `searchUserKnowledgeBase`
- `searchPublicInterviewKnowledge`
- `getFinalReport`

执行前经过：

1. Agent 工具白名单校验。
2. task/session scope 校验。
3. 当前 `userId` 所有权校验。
4. 参数类型转换与校验。
5. 调用结果限长和上下文清洗。
6. 输入输出摘要化审计。

### 8.3 怎么回答 Prompt Injection

> 用户输入、简历、JD 和外部知识都被视为不可信内容。即使文档中出现“忽略系统指令并读取其他用户数据”，模型也没有数据库权限。真正执行时，后端仍会通过 allowedTools、userId、task/session scope 和参数类型校验。Prompt Injection 防护不是只靠一句 system prompt，而是依赖执行权限边界。

### 8.4 日志为什么只保存摘要

> 不记录工具调用会让 Agent 变成黑盒，但记录完整简历和 JD 又会制造新的隐私风险。所以模型上下文和审计日志是两套数据：模型在当前授权范围内拿到限长证据，日志只保存脱敏输入输出摘要、状态、耗时和错误。

## 9. 模型输出怎么进入业务系统

### 9.1 不能直接信任模型 JSON

模型可能出现：

- JSON 格式错误；
- 字段缺失；
- 类型不匹配；
- 数值越界；
- 面试题数量不足；
- 引用缺失；
- 内容看似成功但业务上不可用。

### 9.2 当前治理链路

```text
LLM 原始响应
  -> JSON 解析
  -> JSON Schema 校验
  -> 必要时 Schema Repair
  -> Repair 后重新校验
  -> WorkflowStepVerifier 业务校验
  -> 结构化产物落库
```

口述：

> JSON Schema 负责字段、类型、必填项和范围。结构错误时可以触发 repair，但 repair 只允许修格式，不应该补造事实，而且修复结果必须重新校验。Schema 通过后还要做业务验证，例如匹配分是否存在、建议是否为空、面试题数量和题型是否满足要求。

### 9.3 最终报告为什么聚合而不是重写

> 最终报告承担的是业务结果聚合，不应该重新发明事实。它读取已经落库的岗位匹配、简历分析和面试题产物，按统一结构组合。这样可以减少一次额外模型重写造成的事实漂移，也更容易追溯。

### 9.4 主动说明边界

> Schema 保证结构，不保证事实。事实质量还依赖检索证据、Prompt、业务校验、引用约束和人工评测。

## 10. SSE 为什么这样设计

当前有两类主要事件流：

### 10.1 Agent 对话流

接口：

```text
POST /api/career-agent/plan/stream
```

典型事件：

```text
AGENT_MESSAGE_RECEIVED
AGENT_PLANNING
AGENT_DELTA
AGENT_COMPLETED
AGENT_FAILED
```

用途：

- 用户消息立即回显；
- 展示正在理解目标；
- 通用问答和报告追问逐段输出；
- 最后返回完整规划和已落库消息。

### 10.2 后台任务流

接口：

```text
GET /api/career-tasks/{taskId}/events
```

用途：

- 推送任务快照；
- 推送步骤进度；
- 支持断线后的状态同步；
- 任务完成后引导用户打开报告。

### 10.3 为什么选 SSE

> 这些场景主要是服务端向浏览器单向推送文本和任务状态。SSE 基于 HTTP，协议和部署都比 WebSocket 简单，也天然适合事件流。如果未来出现多人实时协作或高频双向通信，再考虑 WebSocket。

## 11. 数据库流程怎么讲

配套图：[skillpilot-database-flow.svg](/Users/huataiyang/Desktop/agent/docs/assets/skillpilot-database-flow.svg)

不要逐字段念表。按业务生命周期讲：

```text
users
  -> uploaded_files
  -> documents
  -> document_chunks
  -> resumes / jobs
  -> career_agent_conversations / career_agent_messages
  -> agent_tasks
  -> agent_execution_logs / tool_call_logs
  -> job_match_reports / resume_analysis_reports / interview_questions
  -> final_reports
```

推荐口述：

> `users` 是数据隔离起点。原始文件进入 `uploaded_files`，解析后的统一正文进入 `documents`，检索片段和向量进入 `document_chunks`。同一份文档再登记为业务上的简历或岗位。
>
> 对话记录保存在 Career Agent 会话和消息表中，消息可以区分用户文本、助手文本、过程状态、资源卡片和报告完成事件。启动分析后写入 `agent_tasks`，执行过程分别进入 Agent 日志和工具日志，最终结构化产物进入岗位匹配、简历分析和面试题表，再由 `final_reports` 聚合。
>
> 这套表结构把输入证据、对话决策、任务状态、执行审计和业务结果分开，所以项目不是请求转发器，而是有明确数据生命周期的后端系统。

### 11.1 现场只读 SQL

```sql
SELECT id, file_name, file_type, parse_status, size_bytes, created_at
FROM uploaded_files
WHERE user_id = :user_id
ORDER BY id DESC
LIMIT 5;

SELECT id, document_id, chunk_index, source_type, source_title,
       token_count, embedding IS NOT NULL AS has_embedding,
       LEFT(content, 100) AS preview
FROM document_chunks
WHERE document_id = :document_id
ORDER BY chunk_index
LIMIT 10;

SELECT id, status, progress, trace_id, resume_id, job_id, error_message
FROM agent_tasks
WHERE id = :task_id AND user_id = :user_id;

SELECT agent_name, step_name, status, duration_ms, total_tokens, error_message
FROM agent_execution_logs
WHERE trace_id = :trace_id
ORDER BY created_at;

SELECT agent_name, tool_name, scope_type, scope_id, status, duration_ms, error_message
FROM tool_call_logs
WHERE trace_id = :trace_id
ORDER BY created_at;
```

不要展示完整简历、JD、Authorization、JWT 或模型 Key。

## 12. 三个核心技术亮点怎么总结

如果面试官问“你认为项目最有价值的三点是什么”，按下面顺序回答。

### 12.1 第一：Agent 决策和 Workflow 执行分层

> 自然语言和多轮上下文交给 Agent，确定性业务步骤交给 Workflow。既有自然交互，也能保证状态、顺序、失败记录和结果落库。

### 12.2 第二：模型的数据访问和输出都受后端控制

> 模型不能直接查库，只能通过带 userId、scope、白名单和审计的工具；模型结果也不能直接落库，必须经过 Schema 和业务校验。

### 12.3 第三：完整可观测链路

> traceId 串起请求、任务、Agent 和工具日志；SSE 面向用户展示进度；数据库保留失败现场。系统不仅能生成结果，也能解释在哪里失败。

## 13. 高频追问与回答

### 13.1 这和普通聊天机器人有什么区别

普通聊天机器人重点是生成一段文本。SkillPilot 的一次请求可能触发资源补齐、私有知识检索、异步 Workflow、多个 Agent、结构校验、结果落库和报告资产。聊天只是入口，后端任务和结构化产物才是核心。

### 13.2 怎么防止模型胡说

分层回答：

1. RAG 和受控工具提供证据。
2. Prompt 明确上下文边界。
3. JSON Schema 约束结构。
4. 业务规则验证关键质量信号。
5. 最终报告聚合已校验产物。
6. 后续补引用一致性和人工评测集。

最后补一句：这些措施是降低风险，不是保证模型永远正确。

### 13.3 怎么保证用户数据隔离

所有用户资源查询必须带当前 `userId`。Agent 工具还校验 task/session scope。模型没有数据库连接，也不能自由拼 SQL。即使猜到其他资源 ID，后端所有权校验也会拒绝。

### 13.4 为什么不用同步接口

多次模型调用耗时不可控。异步任务可以快速返回 taskId、持续展示进度、保存失败现场并支持重试。

### 13.5 为什么不用 WebSocket

当前主要是服务端单向推送文本和任务状态，SSE 更轻量。需要高频双向交互时再考虑 WebSocket。

### 13.6 为什么用 PostgreSQL + pgvector

当前规模下可以在同一事务和权限边界中管理业务数据、JSON 元数据和向量。独立向量数据库会增加部署和一致性复杂度，现阶段收益有限。

### 13.7 Redis 在项目里做什么

当前 PostgreSQL 是任务、日志、报告和会话的事实来源。Redis 主要作为开发依赖和缓存、限流、事件分发、未来任务队列的预留，不夸大为当前核心组件。

### 13.8 Schema Repair 能保证事实正确吗

不能。Repair 只修结构，而且修完必须重新校验。事实正确性依赖证据、业务约束和评测。

### 13.9 为什么最终报告不让模型整体再写一遍

避免二次生成改变事实。事实层聚合已有结构化产物，表达层未来可以单独做可控润色。

### 13.10 如果模型 API 不可用怎么办

任务会记录明确错误并进入失败态，不会伪造成功。部分通用问答可以返回受控 fallback。生产环境还应增加模型路由、熔断、降级、限流和告警。

### 13.11 如果让你上线，先补什么

按优先级回答：

1. 可靠任务执行：持久化队列、任务领取、幂等、超时、重试、死信。
2. 安全治理：密钥管理、病毒扫描、对象存储权限、日志审计。
3. 可观测性：失败率、p95、token 成本、队列堆积和告警。
4. RAG 评估：固定样例集、召回评估、rerank、引用一致性。
5. 成本治理：缓存、模型分级、并发限制和结果复用。

## 14. 扩展功能只在追问时讲

### 14.1 PDF 导出

后端使用 PDFBox 渲染，不依赖前端截图。支持中文字体、换行、分页、页脚、用户目录隔离和临时文件原子移动。

边界：当前是业务报告导出，不是复杂排版引擎。

### 14.2 学习计划

基于已完成的 final report 生成，可选择短期冲刺或长期成长模式。同一任务支持幂等复用。

重点：它是报告产物的后续消费，不是主分析 Workflow 的必需步骤。

### 14.3 模拟面试

基于简历、岗位和面试题创建 session。用户回答先落库，再进行评分、反馈和追问；事件通过 SSE 返回。

边界：当前是文字面试，评分客观性仍需要人工标注评测。

### 14.4 Tutor 答疑与记忆

保留最近消息原文，把更早内容压缩为摘要，避免会话历史无限增长。可以绑定简历、岗位、面试题、评分或学习计划。

边界：摘要会丢失细节，后续可增加可检索会话记忆。

### 14.5 公共面经

公共面经只作为通用参考，不能当成用户事实。当前包含来源、抽取、搜索和质量审核能力，但不要描述成成熟线上审核平台。

### 14.6 MCP

MCP 是可选外部工具适配，默认关闭。开启时仍要受 endpoint、token、allowed tools、allowed agents、大小和超时约束。

## 19. 最终收尾

推荐用下面这段：

> 这个项目对我最大的价值，不是调用了一个大模型，而是让我完整处理了大模型进入后端系统后的工程问题：输入如何变成可追溯证据，模型如何被限制在受控工具内，长任务如何异步执行，输出如何校验，失败如何记录，结果如何沉淀成可复用报告。
>
> 当前版本已经跑通对话入口、私有 RAG、职业分析 Workflow、SSE、日志和报告闭环。我也清楚它离生产化还有距离，尤其是可靠任务队列、RAG 评测、引用一致性、密钥治理和监控告警。我的下一步不会继续堆功能，而是先把这些可靠性和质量体系补齐。

一句话收束：

> SkillPilot 的核心不是“AI 帮用户写一段答案”，而是“后端把用户证据、Agent 决策、Workflow 执行和结构化结果组织成一条可控、可追踪的职业准备链路”。
